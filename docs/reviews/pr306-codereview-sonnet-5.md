## HIGH severity findings

1. **`/pm_dobot`'s "view-only phase 1" claim does not hold — `/compact` is a free-text injection channel into the viewed session.**
   `apps/web/src/components/action-bar/CommandsSheet.tsx:67-70` renders the `/compact` button *unconditionally* — it sits outside the `{!restricted && <>...}` block (lines 49-66) that hides `/new`/`/resume`/`/branch`/`/rename` for non-default sessions. Tapping it opens `CompactConfirmModal` → `CompactFocusModal` (`apps/web/src/components/action-bar/CompactModals.tsx:38-52`), a `<textarea>` with no content restriction, or `ContinuityNotesModal` (lines 61-75), another free-text `<textarea>`. Both submit through `handleCompact()` (`ActionBar.tsx:163-182`, note the comment at 167-169 explicitly acknowledging "/compact reaches this with a non-default target"), which calls `sendCompactCommand(message, restrictedSession)` → `POST /api/terminal/compact`. Server-side, `apps/server/src/routes/terminal/slash-commands.ts:132-151` resolves the target session via `resolvePaletteTarget` (accepts any existing session by design, per the msg-1188 comment at lines 56-71) and sends the **entire `message` string literally**: `execFileAsync("tmux", ["send-keys", "-t", ..., "-l", "--", message])` + `Enter`. The `/compact ` prefix (`ActionBar.tsx:517`) is client-side cosmetic only — nothing on the server validates that `message` starts with `/compact`. Net effect: opening `/pm_dobot` and tapping Compact → Prompt for Continuity (or Compact Focus) lets a user type arbitrary text that is typed, keystroke-for-keystroke, into the pm-dobot tmux pane. This resolves the prior swarm round's dispute (codex-5.4 HIGH vs. codex-5.5/cursor-5.3 "safe by design") — codex-5.4 was right, and the other two reviewers evidently didn't trace the free-text modal path. Suggested fix: for restricted (non-default) sessions, either strip the free-text fields from `CompactConfirmModal`/only allow "Compact Now" with a fixed message, or enforce server-side that `message` for a client-picked session is drawn from a fixed allowlist (e.g. exactly `/compact` or `/compact <server-templated string>`, never arbitrary user text). Add an integration test asserting `/pm_dobot`'s compact path cannot deliver attacker-chosen text to the session.

## MEDIUM severity findings

1. **MarkdownViewer frontmatter heuristic still over-strips and under-recognizes (unresolved from the prior round's M2, not touched by R1).**
   `apps/web/src/components/MarkdownViewer.tsx:39-40`: `hasYamlShape = nonEmptyLines.every(line => /^\s|^#|^[A-Za-z0-9_.$-]+\s*:(\s|$)/.test(line))`.
   - Over-strip: `^\s` matches any line beginning with whitespace regardless of content, so `---\n  npm install\n---\n# Heading` is wrongly treated as frontmatter and the `npm install` line disappears into the collapsed metadata pill instead of rendering as document body.
   - Under-recognize: a root-level indentless YAML sequence item like `- cpc` (as in `tags:\n- cpc`) matches none of the three alternatives (no leading whitespace, no `#`, no `key:`), so `hasYamlShape` is false and valid frontmatter leaks into the visible rendered document instead of being collapsed.
   Both bugs are in the same regex/line and were flagged by 2/6 external reviewers last round in opposite directions; neither was addressed by the R1 commit (which only touched app-routing.ts, prs.ts, sessions.ts, App.tsx per the diff). Suggested fix: replace the partial-grammar heuristic with a minimal real frontmatter/YAML-shape check (accept indentless `- ` sequence lines; require the block to contain at least one top-level `key:` line, not just first-char whitespace, before accepting continuation lines), and add regression tests for both directions.

## LOW severity findings

1. `apps/server/src/routes/terminal/sessions.ts:54-65` — the first-pane-wins pipe-spoof guard is currently correct (session names can't contain `|` per `SESSION_NAME_RE`, so a real session's row always sorts before a `real|x`-named impostor's), but correctness depends on the undocumented tmux implementation detail that `list-panes -a` output is ordered by byte-wise session-name comparison. Not exploitable today; worth a defensive comment noting the dependency, or a test that doesn't rely on ordering.
2. `apps/server/src/routes/prs.ts` `scanRepos()` batches git subprocess calls in groups of `GIT_SCAN_CONCURRENCY` (8) via `Promise.all` — one hung candidate stalls its whole batch up to `GIT_TIMEOUT_MS` (5s), giving a worst case of `ceil(N/8) * 5s` for a cold scan. Doesn't block the event loop (async), but slows `discoverRepos()`/`currentBranchScope()` callers. Fine as a follow-up (e.g. `Promise.allSettled` with continuous scheduling instead of fixed batches).
3. `apps/server/src/routes/prs.ts` `PrPoller.start()` calls `void this.pollOnce()` on each interval tick with no `.catch()`. `runPollOnce()` can't currently reject in practice (every internal call is try/caught), so this is latent rather than live, and pre-existing (confirmed present before the R1 async conversion) — not a regression from this diff, but worth a top-level `.catch()` for defense-in-depth, plus a test exercising the actual-rejection path (currently only the backoff/no-op and catch-per-repo paths are tested).
