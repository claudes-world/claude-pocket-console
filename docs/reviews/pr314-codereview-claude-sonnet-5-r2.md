## MEDIUM severity findings

1. **`apps/web/src/components/action-bar/ActionBar.tsx:582-621,948-960` — Share sheet silently no-ops for up to 30s if reopened mid-publish.** `shareInFlightRef` is a single unscoped boolean; `handlePublishShared` bails at its first line (`if (!viewingFile || shareInFlightRef.current) return;`) with zero UI feedback if a prior publish (up to a 30s timeout) is still in flight. The "open Share" handler resets `shareUrl`/`shareError`/`shareLoading` and bumps `shareSeqRef` on reopen but never checks/resets `shareInFlightRef`, so a user who closes the sheet mid-publish and reopens it gets tappable-but-dead buttons until the stale request's own `finally` clears the ref. The existing regression test (`ActionBar.test.tsx:318-353`) resolves the first request *before* the second click, so it never exercises this path and currently gives false confidence. Fix: scope the in-flight guard per file path (mirror `readingListInFlightRef`'s `Set<string>` pattern) or abort the previous request via `AbortController` on reopen/close.

2. **`apps/web/src/components/action-bar/ActionBar.tsx:148-204` — reading-list check/save share one seq counter, causing a spurious "unsaved" flash on an unrelated global event.** `readingListCheckSeqRef` is bumped on every re-run of the file-viewing effect, including re-runs triggered only by `readingListRefreshVersion` (fired globally by `emitReadingListChanged()` whenever *any* reading-list item is deleted elsewhere, since `ReadingList.tsx` is always mounted). If a save for file A is in flight when an unrelated item is deleted, the resulting seq bump makes A's own in-flight save's completion check (`seq === readingListCheckSeqRef.current`) fail, so `setReadingListSaved(true)` is skipped even though the save succeeded server-side and `readingListInFlightRef`/`setReadingListSaving(false)` were already cleared in `finally`. The button briefly re-renders as unsaved/tappable; a second tap is harmless (server does `ON CONFLICT ... DO UPDATE`) but redundant, and the UI is momentarily wrong. No test covers this (`READING_LIST_CHANGED_EVENT` / `dispatchEvent` doesn't appear in `ActionBar.test.tsx`). Fix: only bump `readingListCheckSeqRef` on an actual path change; use a separate version/seq for refresh-triggered re-checks.

3. **`apps/web/src/components/action-bar/ActionBar.tsx:560-581` — round-1 M6 only half-fixed.** The R2 commit (`c923498`) added correct return-to-pending UI restoration for both audio and reading-list flows, but `handleSaveToReadingList`'s `haptic.success()/error()` and `setStatus(...)` calls (lines 570-571, 575-576) still fire unconditionally on settle, not gated by the same `seq === readingListCheckSeqRef.current` guard that protects `setReadingListSaved`. Contrast with the sibling `handlePublishShared`, where the equivalent haptic/status calls are correctly inside the seq guard. A late-resolving save for file A can still pop a stale status/haptic while the user is now viewing/saving B. Fix: move the haptic/status calls inside the existing seq check.

4. **`apps/server/src/routes/prs.ts:517-550` (`readRepoIcon`) — round-1 M1 still present.** Unchanged `existsSync`→`realpathSync`→`statSync`→`readFileSync` ad hoc chain, not routed through the hardened `openAllowedForRead`. Confirmed lower-severity than H1's vector (`repoPath` originates only from server-side `discoverRepos()` scanning a fixed, non-world-writable `~/code`, not client input), but it's the same TOCTOU shape the hardened primitive exists to close and the size-cap check still precedes the read it's meant to bound. Fix: route through a pinned-fd read; add a symlink-swap regression test.

5. **`apps/server/src/routes/share.ts:6,124` — round-1 M2 still present.** `share.ts` reuses the broad `ALLOWED_FILE_ROOTS` (includes `/tmp` and other lanes' working dirs) to gate what can be published to a public/private URL. A narrower `ALLOWED_SHARE_ROOTS` was recommended and not introduced.

6. **`apps/server/src/routes/prs.ts:663-665` (`GET /issues`) — round-1 M5 still present.** `catch (err) { ... err.message }` returns the raw `gh` CLI stderr straight to the browser (via `execGh`), which can leak local paths/config context.

7. **`apps/server/src/routes/prs.ts:638-667` — round-1 M7 still present, confirmed independently exploitable.** `force=1` on `/api/prs/issues` bypasses the 5-minute cache with no server-side rate limit beyond client-side `refreshing` React state (trivially bypassed by calling the endpoint directly). `POST /refresh` has the same shape for PR polling. Repeated calls can burn the `gh` CLI's GitHub API rate limit across every discovered repo. Fix: minimal per-repo cooldown that ignores `force` if the cache is younger than N seconds, or a global debounce.

8. **`apps/server/src/routes/share.ts:139-160` — round-1 M8 still present.** The 50MB `fstat`-based size check happens once before the copy loop; the loop itself has no running byte cap, so a file that grows after the `fstat` check could still be copied past the limit before anything downstream (e.g. `publish-shared`'s own limits, if any) catches it.

9. **`apps/server/src/routes/prs.ts:669` (`GET /icons`) — round-1 M11 still present.** Handler remains fully synchronous; `readRepoIcon` is built on sync `fs` calls, up to 14 candidates × N repos on a cache miss, blocking the event loop.
