## HIGH severity findings

**1. Merge with `dev` will conflict on security-critical code; naive resolution reintroduces a fixed FIFO-DoS and can drop the session-picker fix.**
`git merge-base origin/dev origin/dev-cpc-features` shows `dev` has 3 commits (PR #299, "round-1/round-2 super-swarm hardening") not present on `dev-cpc-features`. Two of the touched files are ones this PR's new code depends on:

- `apps/server/src/lib/path-allowed.ts`: on `dev`, `openAllowedForRead` was hardened (PR #299 round-2) to open with `O_RDONLY | O_NONBLOCK` and `fstat`-reject anything that isn't a regular file or directory, with an explicit comment: *"`/tmp` is a world-writable read root, so a client-controlled path can name a FIFO. Opening a FIFO for read with the plain `"r"` flag blocks the libuv threadpool until a writer shows up — a cheap DoS."* `dev-cpc-features` still has the pre-fix version (`open(resolve(absPath), "r")`, no fstat gate) — confirmed via `git show origin/dev-cpc-features:apps/server/src/lib/path-allowed.ts`. This PR adds `apps/server/src/routes/share.ts`, whose `/publish` handler does exactly `openAllowedForRead(resolve(body.path), ALLOWED_FILE_ROOTS)` where `ALLOWED_FILE_ROOTS` includes `/tmp` (confirmed by reading both files at the `dev-cpc-features` ref). Any process running as the same `claude` UID — which on this host includes every other WorldOS lane/agent — can `mkfifo` a path under `/tmp` and hand it to `POST /api/share/publish` (or any other `openAllowedForRead` caller reachable with a `/tmp` path) to block a threadpool thread indefinitely on the un-patched branch; with the default 4-thread libuv pool, a handful of concurrent requests exhausts it and stalls every filesystem-touching route server-wide.
- `apps/web/src/App.tsx`: `dev` reworks the same lines this PR touches — it replaces the inline `sessionList.length > 0` SessionPicker guard with `resolveSessionPickerProps(...)` (fixing a round-2/PR#299 bug where the picker was hidden entirely whenever `/api/terminal/sessions` failed while a stale session was active, stranding the user), on the *same* state-declaration block and the *same* `FileViewer`/`Links` render lines this PR's reading-list-open sequencing fix touches (`fileOpenRequest`/`fileOpenRequestRef`/`handleFileViewChange`/`openFileFromReadingList`). These are two independent, already-reviewed fixes landing on overlapping lines — resolving the conflict by mechanically picking one side loses the other.
- `apps/server/src/index.ts`: `dev` replaces the `shareRoute` import line with an unrelated `isAssetLikePath` import at the same line number (`dev` doesn't have `share.ts` yet) — a direct textual conflict on import resolution, low risk in isolation but confirms the conflict is real, not a false positive from GitHub's mergeability check.

Suggested fix: before merging #314, merge/rebase `dev-cpc-features` onto the current `dev` tip and manually verify (a) `path-allowed.ts` keeps the `O_NONBLOCK` + fstat gate, ideally with a regression test asserting `openAllowedForRead` on a FIFO path returns `denied` instead of hanging, and (b) `App.tsx` keeps both `resolveSessionPickerProps` and the reading-list `fileOpenRequest` sequencing. Do not let GitHub's auto-merge or a quick "resolve conflicts" pass stand in for that verification.

**2. `share.ts`'s TOCTOU fix closes the source-read race but reopens a narrower one at the staged-file handoff.**
`apps/server/src/routes/share.ts` (dev-cpc-features): the source file is read through a pinned fd (`openAllowedForRead` → `handle.stat()` → `handle.createReadStream()`), which genuinely closes the original check-then-open race for the *source*. But after streaming into `fs.mkdtemp(join(tmpdir(), "cpc-share-"))` and closing that handle, the route calls `execFileAsync(PUBLISH_SHARED, [...args, stagedPath], ...)` — handing `publish-shared` the staged file **by path string**, with no fd held on it anymore. `publish-shared` (external, not in this repo) necessarily reopens that path by name, reintroducing a check-then-act window one hop downstream. `mkdtemp`'s default `0700` mode limits this to same-UID actors, but every WorldOS lane on this host runs as the same `claude` account, so "same UID" is not a meaningfully smaller threat population here. The PR's claim to have closed "the" TOCTOU is accurate for the read step only. Confirm whether the already-filed follow-up (#308, referenced in the integration notes) covers this exact staged-file gap, or file a new one scoped to the CPC-side staging handoff, not just the external helper's own open-time race.

## MEDIUM severity findings

**3. `handleSaveToReadingList`'s status/haptic feedback isn't seq-guarded, unlike the sibling share flow.**
`apps/web/src/components/action-bar/ActionBar.tsx` (dev-cpc-features): traced the exact scenario the "per-path save set" fix targets — save file A (bumps `readingListCheckSeqRef` to `sA`, adds A to `readingListInFlightRef`), navigate to B before A resolves (bumps seq again, correctly clears the Saving/Saved *button* state for B via the `pathChanged` branch), save B (bumps seq again), then A's save resolves late. The `if (seq === readingListCheckSeqRef.current) setReadingListSaved(true)` guard correctly suppresses marking A "Saved" while B is being viewed, and the `finally` block's `readingListInFlightRef.current.delete(path)` correctly runs unconditionally, so a later re-save of A is never permanently blocked — the specific race this fix was built for is closed.
However, the surrounding `haptic.success()/haptic.error()` and `setStatus(...)` calls in the same try/catch are **not** guarded by the seq check — they fire unconditionally on every settle. `status` is a single shared `useState` string rendered by one `StatusLine` for the whole ActionBar (used by audio, git, reading-list, etc. alike). So in the traced scenario, A's save resolving while the user is now viewing/saving B pops "Saved to reading list" (or a `Failed: ...` message) attributed to whatever the user is doing with B at that moment — a stale, mismatched status flash. Contrast with `handlePublishShared` in the same file, which guards its entire success/failure branch (including `setStatus`) behind `if (seq !== shareSeqRef.current) return;` — the reading-list save should adopt the same shape for consistency and to close this residual UX-correctness gap.

**4. `share.ts` reuses the broad view-only `ALLOWED_FILE_ROOTS` to gate what may be published to a public URL.**
`ALLOWED_FILE_ROOTS` (`path-allowed.ts`) is documented as the *read/view* allowlist and explicitly includes `/tmp` and `/home/claude/.worldos/lanes` ("legacy lane workspaces — other agents' working state"), with a comment noting it's a superset of the write roots specifically for viewing. `share.ts` passes this same list to gate publication to `shared.claude.do` (public or Cloudflare-Access-gated private). Anything viewable in CPC — including another agent lane's working files under `.worldos/lanes` — can be pushed to an external URL. Recommend a narrower `ALLOWED_SHARE_ROOTS` distinct from the viewer's allowlist.

**5. `readRepoIcon()` (prs.ts) uses a split existsSync→realpathSync→statSync→readFileSync sequence instead of the established pinned-fd pattern.**
This is the same TOCTOU shape `openAllowedForRead` was built to close, just not using that primitive — four independent syscalls against the same path string, each re-resolving symlinks at call time. Exploitability is low today (`repoPath` only ever comes from `discoverRepos()` scanning `~/code`, not client input, and `~/code` isn't world-writable), but it's inconsistent with the pattern the rest of this integration group converged on, and it's untested: `prs-routes.test.ts`'s `/icons` tests cover candidate ordering, the 64KB cap, and MIME mapping, but no case exercises the symlink-escape guard (`relative(realRepoPath, realCandidatePath)` starting with `..`). Add that test and consider routing through `openAllowedForRead`.

**6. `/api/prs/issues?force=1` bypasses the 5-minute cache with no independent server-side rate limit.**
`force=1` unconditionally skips the `issueCache` TTL check. The `repo` param is validated against `discoverRepos()` (closing arbitrary-string fan-out), but a scripted authenticated client can still loop `force=1` against a real discovered repo with no server-side throttle beyond the client's own `refreshing`-disabled UI courtesy, burning `gh` subprocess spawns and GitHub API quota. Narrower than the "unbounded fan-out" risk the integration notes claim is closed, since it requires valid auth and a real repo name — but it is a residual gap the notes don't call out.

## LOW severity findings

**7. Icon MIME trust is extension-based, not content-sniffed.** `ICON_MIME` in `prs.ts` assigns `image/svg+xml`/`image/png`/`image/x-icon` purely from the candidate filename's extension. Currently inert — the only consumer (`PrTicker.tsx`) renders `repoIcons[...]` exclusively via `<img src={...}>`, which doesn't grant SVG script execution — but there's no guard against a future `dangerouslySetInnerHTML`/`<object>`/`<iframe>` consumer silently turning this into stored XSS. Worth a one-line comment at the `ICON_MIME` definition.

**8. Audio deep-link's uppercase-extension fallback is a known, accepted limitation** (`audio.ts`, documented inline): `note.MD` beside `note.mp3` is never discovered because the derivation is a literal lowercase `.md` replace; on a case-sensitive filesystem this silently falls back to the audio link instead of the doc. Acceptable given doc-writing tooling here always emits lowercase, but flag if that assumption ever changes.

**9. `env: {...process.env, ...}` in `share.ts`'s `execFileAsync` call** spreads the full parent environment (including `TELEGRAM_BOT_TOKEN`/`OPENAI_API_KEY`) into the `publish-shared` child. This is not a new exposure — it's the existing trust model every host-tool shellout in this codebase already uses (e.g. `audio.ts`'s `curl` calls inherit the same env implicitly by omitting the option) — but since this is the first route to spread `process.env` *explicitly*, a future reader could mistake the explicit spread for a deliberate widening. No action needed beyond awareness.

## Cross-cutting observations

- **Seq-token convergence.** The stale-async-response guard pattern (a monotonic ref bumped on every new request, compared before applying a late response) is now independently reimplemented five times: `audioCheckSeqRef`/`audioInFlightRef` (ActionBar audio), `shareSeqRef` (ActionBar share), `readingListCheckSeqRef`/`readingListInFlightRef` (ActionBar reading-list), `fileOpenRequest.sequence`/`fileOpenRequestRef` (App.tsx FileViewer), and `issuesLoadSeqRef` (PrTicker issues). Each is correctly applied where traced (see finding #3 for the one exception found), but the duplication is real — worth extracting a small `useLatestRequest`/`useAbortableEffect` hook before a sixth copy gets pasted in with a subtly different bug.
- **The diff under review is not the merge target.** Everything above about `dev`'s divergence is invisible from the diff alone — the diff is `dev-cpc-features` vs. its own merge-base, not vs. current `dev`. Any future re-review of this PR must diff against current `dev`, not re-read this same diff, or the conflict-adjacent findings above will look "already fixed" when they're actually just absent from the compared range.
- **`App.tsx`'s FileViewer callback-ownership fix (`fileOpenRequest`/`fileViewerSequence`/`fileOpenRequestRef`) independently verified sound.** Traced multiple rapid reading-list opens (A→B→C in quick succession): each `handleFileViewChange`/`handleFilePathChange` closure captures its own request's sequence number at creation time and compares against the always-current `fileOpenRequestRef.current.sequence` at call time, so only the latest FileViewer instance's callbacks can ever match and apply — every older instance's late callback correctly no-ops regardless of how many jumps occurred in between. Grepped the whole file for `setViewingFile`/`setCurrentFolder`: the only unwrapped call is `openFileFromReadingList`'s synchronous `setViewingFile(null)` reset, which isn't passed to a child as a callback, so nothing bypasses the guard.
- **Reading-list server route (`reading-list.ts`) is unchanged by this PR** but is the direct dependency of the new `ReadingList.tsx` UI and `action-bar/api.ts` calls; it correctly re-validates `path` via `isPathAllowed(normalizedPath, ALLOWED_FILE_ROOTS)` at `/save` time, uses parameterized queries throughout, and doesn't independently open the file (so it isn't exposed to the FIFO issue in finding #1 the way `share.ts` is) — no action needed, noted for completeness since the brief asked about Reading List's interaction with existing storage state.
