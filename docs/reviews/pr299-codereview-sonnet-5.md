# Phase Super-Swarm Review -- PR #299 (head 5cf377e)

## HIGH severity findings

**H-1. `getPaneDimensions` runs a synchronous, blocking tmux call inside the 500ms terminal poll loop, and #291 directly multiplies how many of these can run concurrently.**
`apps/server/src/routes/terminal-ws.ts:20-36` (call site `terminal-ws.ts:272` inside `sendPaneContent`, scheduled by `setInterval(sendPaneContent, 500)` at `terminal-ws.ts:328`). `getPaneDimensions` uses `execFileSync("tmux", ["display-message", ...], { timeout: TMUX_TIMEOUT_MS })` — a **synchronous** child-process call that blocks Node's single-threaded event loop for its full duration (up to the 5s `TMUX_TIMEOUT_MS` if tmux is slow/wedged) on every poll tick, for every open WS connection. Before this PR, the realistic ceiling on concurrent pollers was "however many browser tabs happen to be open on the one shared session." #291's whole point is to make opening multiple simultaneous terminal connections a normal, encouraged workflow (SessionPicker, deep links, "nudge a stuck lane" — Liam voice msg 1188), so this release turns a latent scalability wart into an availability risk: N concurrently-viewed sessions means N synchronous stalls competing for the same event loop, each capable of freezing *all* other traffic (auth, file reads, other WS sockets) on the server for up to 5s if any one tmux target is slow. The file already uses the async `execFileAsync` pattern everywhere else in the same module (`has-session`, `applyFitResize`) — `getPaneDimensions` is the one holdout. Suggested fix: convert it to `execFileAsync` (or drop it in favor of parsing dimensions out of the same `capture-pane`/`list-panes` calls) so it can't block the loop.

## MEDIUM severity findings

**M-1. `reading-list.ts` and `telegram.ts` were not migrated to the race-safe read path, even though they consume the newly-widened (world-writable-inclusive) `ALLOWED_FILE_ROOTS`.**
`apps/server/src/routes/reading-list.ts:38`, `apps/server/src/routes/telegram.ts:12`. Both still call the by-name `isPathAllowed(path, ALLOWED_FILE_ROOTS)` (check-then-use) rather than `openAllowedForRead`. Today this is benign: neither route ever opens the file's *content* — `reading-list.ts` only stores/queries the path string in SQLite, and `telegram.ts` only embeds the path in a Telegram message. But this is precisely the migration the #292 PR set out to complete ("closes TOCTOU on content reads and /search walks"), and it silently stopped short of every consumer of the now-`/tmp`-including root list. If either route grows a "preview" or "send file content" feature later (plausible — reading list already looks like a bookmark-then-open flow), it will inherit an unguarded check-then-use pattern against a world-writable root with no obvious signal that it needs the fd-pinned helper instead. Worth a tracking follow-up so the migration doesn't quietly regress.

**M-2. SPA fallback turns stale-cached asset requests into a silent 200 `text/html` instead of a clean 404.**
`apps/server/src/index.ts:203-223`. The new catch-all serves `index.html` for any GET that isn't `/api/*` or `/ws/*` and wasn't matched by `serveStatic`. This is normally what you want for client routes, but it also now swallows requests for **stale hashed asset paths** — e.g. a browser tab left open across a deploy, holding an old `index.html` that references `/assets/app-<oldhash>.js`, which no longer exists after the next `@cpc/web` rebuild. Previously that 404'd cleanly; now it gets a 200 HTML response, and the browser's module loader throws a confusing "expected a JavaScript module script" error instead of a clear network 404, with no service worker or cache-busting in place to force a reload. Directly relevant to the release's own deploy-ordering concern (rebuild web + server, restart) — a user who doesn't reload between the tag and the restart will see this. Not data-destructive, but worth a `Cache-Control: no-store` on `index.html` or an asset-path allowlist (e.g. `/assets/*` returns 404, not the shell) to keep the failure mode obvious.

**M-3. File-search "jump to result" silently drops the multi-session terminal deep link.**
`apps/web/src/components/action-bar/ActionBar.tsx:562` (pre-existing code, newly exposed by #291): `onSelect={(result) => { window.location.hash = \`files&file=...\`; window.location.reload(); }}` overwrites the *entire* hash rather than merging params, unlike the new `onSelectSession` in `App.tsx:2904-2916`, which carefully preserves other hash params when rewriting `&session=`. A user viewing a non-default session (`#terminal&session=do-box--lane-a`), who then opens file search and taps a result, gets reloaded onto `#files&file=...` — the `session=` param is gone, so returning to the terminal tab silently reverts to the default session with no indication the view changed. Low-effort fix: build the new hash by merging into `URLSearchParams` the same way `onSelectSession` already does, instead of replacing the whole hash string.

**M-4. Checked-in docs are stale relative to what this release ships, and the staleness has real operational teeth.**
`apps/server/CLAUDE.md` "File Access" section still lists the pre-#292 5-root list (no `.worldos/lanes`, no `/tmp`, no read/write split) and the "Route Structure" table is missing `sessions.ts`'s new endpoint and the `/cpc-branch` behavior change. More importantly, `docs/guides/deploying.md`'s "Manual Deploy Steps" (`cd ~/code/claude-pocket-console && nohup npx tsx apps/server/src/index.ts &`) describes a flow that no longer matches the real production setup: `cpc.service` (systemd) actually runs the **built** `apps/server/dist/index.js` with `WorkingDirectory=/home/claude/code/claude-pocket-console-prod`. Since #288's `/cpc-branch` fix now reports `process.cwd()` specifically so the UI shows the *deployed* version, an operator who follows the checked-in manual-deploy doc instead of the systemd unit would start a second, untracked process from the wrong checkout directory — directly undermining the accuracy guarantee #288 just shipped. Worth reconciling the doc with the actual `cpc.service` unit before the next person "deploys" from it.

## LOW severity findings

**L-1. `terminal/git.ts` keeps its own hand-duplicated `ALLOWED_ROOTS` array** (`apps/server/src/routes/terminal/git.ts:13-19`) instead of importing `ALLOWED_WRITE_ROOTS`/`ALLOWED_FILE_ROOTS` from `lib/path-allowed.ts`. Currently correct (git commands shouldn't run against `/tmp` or lane workspaces, so the narrower hand-kept list is arguably *right*), but it's explicitly flagged in its own comment as debt pending a shared `lib/allowed-roots.ts` (S-1). Worth linking that follow-up to this release so it isn't forgotten now that there are three divergent root lists (`git.ts`'s own, `ALLOWED_WRITE_ROOTS`, `ALLOWED_FILE_ROOTS`) instead of two.

**L-2. Multi-session terminal + view-only file roots implicitly assume a single authorized user.** `ALLOWED_TELEGRAM_USERS` currently allowlists exactly one ID, so this isn't exploitable today, but both #291 (any allowed user can view any tmux session, including other lanes' live pane content) and #292 (any allowed user can browse `.worldos/lanes` and `/tmp`, i.e. every agent's workspace) have zero per-user scoping baked in. If the allowlist ever grows past one user, both features become a cross-user visibility leak by design, not by bug. Worth a comment or a tracking issue so it's a deliberate decision if/when the allowlist grows.

**L-3. ActionBar deep-link flicker for a literally-named default session** (`apps/web/src/App.tsx:2856-2857`, `ActionBar.tsx:35-38`): a deep link like `#terminal&session=claudes-world` (naming the default session explicitly) renders the *restricted* palette until the `/api/terminal/sessions` fetch resolves and `defaultSession` is known, then flips to the full palette. Already called out and accepted in the code's own comment as a deliberate "pessimistic default is safe either way" tradeoff — flagging only because it's a real, visible one-request-round-trip UI flicker, not because the tradeoff itself is wrong.
